MIT6.5840学习总结

最初接触这个项目是为了学 Go语言——我觉得从项目入手学一门语言是最快的方式。虽然这种做法难以搭建起完整的系统化认知框架，但系统性地学一门语言又太耗时间了，所以权衡下来这是个不错的折中。后来，我在百度找到了这个项目，没想到难度还是挺大的：项目提供了框架、详尽说明和测试用例，我只需实现核心逻辑，但整体工作量还是不小，实际花费的时间也远超预期。

好在也学到了点东西：掌握了 Go 的基础语法，理解了协程、通道和同步机制的使用方式，也对分布式系统有了初步认识，明白了共识算法如何生效、CAP 定理的权衡等。这个项目共包含五个实验，每个实验又分成多个章节。

项目设计特点

这个项目的精妙之一，在于它通过单机模拟的方式，为学习者构建了一个高度逼真且易于调试的分布式系统实验环境。在实际部署中，分布式系统的各个节点通常运行在不同的物理机器上。而在此项目中，多个节点（如多个 Raft 服务器副本、多个 MapReduce Worker）是以多个独立进程的形式，在同一台机器上启动的。

为了模拟真实的网络通信，项目提供了一个封装好的 RPC（远程过程调用）框架。这个框架至关重要，它让我们无需深入底层网络编程（如 Socket 管理、数据序列化与反序列化等），只需像调用本地函数一样定义和调用服务，框架会自动处理进程间的通信细节。

更重要的是，这个 RPC 框架并不仅仅是简单的进程间通信桥梁。为了重现分布式环境的复杂性，它在底层引入了各种网络故障，例如：

  • 网络延迟：随机地延迟 RPC 请求或响应
  • 消息丢失：随机地丢弃 RPC 请求，模拟网络不可靠
  • 重复请求：模拟因超时重传而导致的重复消息

正是通过这些有意的”破坏”，我们的代码才必须被设计成能够容错的——例如，Raft 算法需要处理丢失的 AppendEntries RPC，KV 服务器需要处理重复的 Put 请求。因此，我们虽然身处一个稳定的单机环境，但所编写的每一行代码，都在应对一个充满不确定性的”真实”分布式世界。这种设计使得我们能够将全部精力集中在分布式核心逻辑（如共识、状态复制、容错）的实现上，而非基础设施的构建。

实验概览

MapReduce

要求：实现一个分布式的MapReduce框架。项目中已经编写好了mapreduce函数，我们需要编写协调器（Coordinator）和工人（Worker）程序。协调器负责将任务分发给空闲的工人，并处理故障恢复——当某个工人节点长时间无响应时，协调器需要将其任务重新调度给其他工人。关键在于协调器的状态管理（任务分配、完成状态跟踪）和工人节点的容错机制，最终确保所有任务都能被正确执行和汇总。

测试情景与设计思想

课程自带的测试代码相当漂亮，它通过一组设计严苛的测试案例，从正确性、容错能力到性能，对系统进行了全方位的验证。

1. 功能性测试

  • wc.go：标准词频统计。验证框架在无故障情况下处理经典MapReduce场景的能力
  • indexer.go：倒排索引构建。验证框架处理不同业务逻辑的正确性，测试中间结果的组织方式

2. 容错性测试

  • crash.go：随机崩溃测试。
    • 其Map/Reduce函数有约1/3概率直接退出进程，1/3概率延迟执行，1/3概率正常执行
    • 设计思想：模拟真实的 worker 节点故障。它迫使协调器必须能够检测到 worker 失联（通过任务超时），并将任务重新调度给健康的 worker。通过此测试，证明您的协调器实现了有效的故障恢复机制
  • early_exit.go：Worker 提前退出测试
    • 某些 Reduce 任务会长时间睡眠，模拟 worker 僵死或网络分区
    • 设计思想：测试协调器对”僵尸”任务的处理能力。即使 worker 进程没有崩溃但停止了进度，协调器也应能通过超时机制收回任务并重新分配

3. 正确性与可靠性测试

  • jobcount.go：任务分配计数测试
    • 设计思想：统计任务被执行的次数。即使在无故障情况下，也要确保协调器不会因编程错误（如重复通知完成）而导致同一个任务被多次成功提交。此测试用于验证您实现的任务状态机的严谨性
  • nocrash.go：无崩溃对照测试
    • 保留 crash.go 的结构但禁用崩溃功能，作为对照实验，用于排除因崩溃注入本身带来的干扰，确保系统在正常情况下的结果是正确的

4. 并行性能测试

  • mtiming.go & rtiming.go：Map/Reduce 阶段并行度测试
    • 设计思想：验证 worker 是否能真正并行地执行 Map 和 Reduce 任务，这是 MapReduce 提升性能的关键
    • 并行检测机制：这两个测试使用了巧妙的 nparallel() 函数来检测并行度
      • 当任务开始时，worker 会在磁盘上创建一个以自身进程ID命名的临时文件（如 mr-worker-reduce-）
      • 随后，函数会扫描目录，统计由其他 worker 创建的同类临时文件的数量
      • 返回值就是当前正在并行执行该类型任务的 worker 数量
      • 通过检查 nparallel() 的返回值是否大于1，来验证并行性。如果您的协调器设计得当，能够同时向多个空闲 worker 分发任务，就能通过此测试
    • 补充细节：在 rtiming.go 的 Map 函数中，故意生成了 a 到 j 共10个不同的键，这是为了确保中间结果能被分区到多个 Reduce 任务中，从而为测试 Reduce 阶段的并行性创造前提条件。如果所有键都相同，则只会产生一个 Reduce 任务，无法测试并行

设计与实现

这个实验作为入门项目，核心目的在于理解分布式任务调度与容错的基本思想。实现上主要包含两个组件：

协调器

  1. 状态管理：维护所有Map与Reduce任务的完整状态信息，包括待分配、执行中、已完成三种核心状态。使用互斥锁对共享状态的任何读写操作进行同步，确保在高并发请求下的数据一致性，避免竞态条件

  2. 任务调度：使用带缓冲的Channel（mapTaskChan与reduceTaskChan）构建待分配任务队列，实现生产者-消费者模型。当工人节点通过RPC请求任务时，协调器从对应的Channel中取出任务进行分配

  3. 超时与重试：为每个已分配任务启动超时计时器（默认10秒），当检测到任务超时时，自动将任务状态重置为空闲，并重新加入分配队列，通过重试计数器实现任务重试上限控制（最多3次），防止无限重试

  4. 阶段协调：遵循MapReduce执行模型，确保所有Map任务完成后才启动Reduce阶段，通过维护已完成Map任务计数器（CompletedMapTasks）与总任务数（NMap）的比对来实现阶段转换，保证Reduce阶段输入数据的完整性

工人

  1. 任务获取：启动后进入任务请求循环，定期向协调器请求任务（基础间隔100毫秒），如果未获取到任务则等待2秒再尝试

  2. 任务执行：根据任务类型调用相应的Map或Reduce处理函数，Map任务产生的中间结果键值对，会依据键的哈希值被分区到不同的中间文件中，为后续的Reduce阶段做准备。输出文件遵循统一的命名规范：mr-[Map任务编号]-[Reduce任务编号]，以确保数据能被正确归并与查找

  3. 状态同步：任务完成后通过RPC调用Coordinator.TaskCompleted方法通知协调器，告知任务已完成，确保任务状态及时更新

KV Server

要求：构建一个单点服务的键值存储服务器，支持 Get、Put 和 Append 操作。这个实验的挑战在于处理各种网络故障。在有网络故障时，需要处理诸如：客户端请求超时后重发（可能导致操作被重复执行）、服务器处理成功后但响应丢失、以及因网络延迟导致的乱序请求等问题。核心是实现一套机制来保证操作的幂等性，例如为每个客户端请求分配唯一的序列号，确保即使收到重复请求，服务器的状态也是正确的

设计与实现

这个实验主要是为后续实现分布式键值服务做准备，难度不算大，重点在于单节点服务如何应对并发访问和不可靠网络带来的数据一致性问题——》线性一致性，对操作进行加锁，保证Append重复请求的幂等性。

请求去重与幂等性：为每个客户端分配唯一ID，每次请求附带单调递增的序列号

服务端为每个客户端维护“已处理的最大序列号”，收到请求时，服务端根据序列号判断：如果是已经处理过的请求，直接返回之前缓存的结果，实现幂等性。如果是新的请求，执行操作并更新状态。如果是过时的请求，直接拒绝，避免状态回退

并发控制：由于需要维护客户端序列号状态和执行检查-更新操作，服务端必须在处理写请求时进行加锁，保证这些关键操作的原子性

Raft

要求：基于raft论文，实现raft共识算法，包括领导者选举、日志复制、日志持久化、快照（简化版本），要通过在复杂网络以及并发下的不同测试。核心是实现一个能够在网络分区、节点故障等异常情况下仍能保持一致性的分布式共识协议。

设计与实现

Raft实验是课程中最具挑战性的部分，需要深入理解分布式共识的核心原理。

核心组件

  1. 状态机：每个节点维护当前状态（领导者、跟随者、候选人）
  2. 日志管理：维护操作日志序列，确保所有节点最终一致
  3. 选举机制：实现超时驱动的领导者选举算法
  4. 心跳机制：领导者定期发送心跳维持权威
  5. 日志复制：领导者将日志条目复制到跟随者
  6. 持久化存储：将关键状态（当前任期、投票记录、日志）持久化
  7. 快照压缩：定期创建状态快照以节省存储空间

关键算法

  • 领导者选举：基于随机超时和多数派投票
  • 日志复制：使用PrevLogIndex和PrevLogTerm确保日志一致性
  • 安全性保证：通过选举限制和提交规则防止脑裂和数据丢失
